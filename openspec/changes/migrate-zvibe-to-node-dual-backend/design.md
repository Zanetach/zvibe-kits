## 上下文

当前项目已具备可用的命令形态（setup/config/status 与 agent 启动），但核心实现基于 Bash + AppleScript，随着能力增长在以下方面风险升高：
- 可维护性：命令逻辑、依赖检查、布局编排耦合在单脚本中，演进成本高。
- 稳定性：Ghostty GUI 自动化链路对环境/权限/快捷键高度敏感，失败场景定位成本高。
- 可用性：错误提示偏底层，缺少结构化诊断与可重复修复流程。

本次设计以“保持用户心智稳定”为前提：保留现有命令语义与默认行为，将实现内核升级为 Node/TypeScript，并引入 Ghostty + tmux 双后端模型。

相关角色：
- 终端日常用户：关注启动成功率、命令简洁、配置一致性。
- 维护者：关注代码结构、扩展成本、故障定位与发布演进。

## 目标 / 非目标

**目标：**
- 在不破坏现有命令习惯的前提下，将 CLI 核心迁移到 Node/TypeScript。
- 支持 `ghostty` 与 `tmux` 后端共存，并提供 `auto` 选择策略与可配置降级。
- 明确并固化 Agent 启动策略：`defaultAgent`（无参数）与 `agentPair`（`zvibe code`）。
- 升级 setup/config/status 交互：中文输出、可行动错误、结构化状态结果。
- 构建可演进架构：命令层、配置层、后端层、诊断层职责分离。

**非目标：**
- 不在本变更中新增全新产品命令家族（保持命令面简洁）。
- 不在本变更中重构 agent 本体（codex/claude/opencode）的行为。
- 不承诺一次性消除 Ghostty 自动化限制，仅提供更稳健的回退与诊断。
- 不在本阶段引入复杂云端服务或远程编排能力。

## 决策

### 决策 1：采用 Node/TypeScript 重构 CLI 核心
- 选择：以 TypeScript 为主，构建单一 CLI 入口，替代 Bash 主流程。
- 原因：类型约束、模块化、可测试性和错误建模能力更适合后续扩展。
- 备选方案：
  - 继续 Bash：短期改动小，但长期维护与稳定性上限明显。
  - 改用 Python：可行，但与当前 npm 分发形态与生态衔接较弱。

### 决策 2：后端采用双实现并存（ghostty-applescript + tmux）
- 选择：定义统一 Backend 接口，运行时按配置选择 `ghostty|tmux|auto`。
- 原因：既保留现有 Ghostty 使用方式，又提供 tmux 作为高稳定兜底与性能路径。
- 备选方案：
  - 仅保留 Ghostty：用户体验一致，但脆弱性不可控。
  - 仅保留 tmux：稳定但破坏现有 Ghostty 习惯，不符合目标。

### 决策 3：`auto` 策略默认开启并允许关闭降级
- 选择：默认 `backend=auto`，优先 ghostty，失败后可降级 tmux（受 `fallback` 控制）。
- 原因：在多数用户场景下提升启动成功率，同时保留显式控制权。
- 备选方案：
  - 默认强制 ghostty：可预测但失败率更高。
  - 默认强制 tmux：稳定但违背“优先保留 Ghostty 体验”。

### 决策 4：固化 Agent 启动契约，拒绝隐式猜测
- 选择：
  - `zvibe`（无参数）必须读取 `defaultAgent`；缺失/非法则 fail-fast。
  - `zvibe code` 读取 `agentPair`；CLI 参数优先于配置。
- 原因：减少“为什么启动成这样”的不确定性，提升可解释性。
- 备选方案：
  - 自动猜测可用 agent：看似友好，但行为不稳定、调试困难。

### 决策 5：统一交互与诊断协议
- 选择：
  - 命令统一支持 `--json` 与 `--verbose`。
  - setup/config/status 输出中文并包含“原因 + 修复建议”。
  - 引入标准错误码（如 `E_CONFIG_INVALID`, `E_GHOSTTY_ACCESS`, `E_TMUX_MISSING`）。
- 原因：同时满足人类可读与脚本可读，提升排障效率与自动化能力。
- 备选方案：
  - 仅保留文本输出：易读但不利于自动化与集成。

### 决策 6：分阶段迁移，先等价再优化
- 选择：
  1) 等价迁移命令语义；2) 双后端接入；3) 诊断增强；4) 性能优化。
- 原因：控制风险、减少一次性重写失败概率。
- 备选方案：
  - 一次性大重构：理论更快，实际风险高、回归点多。

## 风险 / 权衡

- [Ghostty 自动化仍受系统权限/GUI 状态影响] → 通过 preflight 检查、错误码和 tmux 降级降低失败率。
- [双后端增加实现复杂度] → 使用统一接口与公共布局模型，避免命令层分叉。
- [迁移期间行为偏差风险] → 设定等价行为验收清单，逐命令回归（setup/config/status/默认启动/code 模式）。
- [用户对新配置字段不熟悉] → 保持旧字段语义可映射，并提供 `config explain` 风格说明（文档/输出）。

## 迁移计划

1. 建立 Node CLI 骨架与配置 schema，保留现有命令入口。
2. 完成 Agent 解析与默认行为迁移，确保 `defaultAgent` 契约生效。
3. 接入 Ghostty backend（行为对齐现有流程）。
4. 接入 tmux backend 与 `auto` 路由、`fallback` 策略。
5. 升级 setup/config/status 的诊断、错误码和中文输出。
6. 运行回归验证并发布。

回滚策略：
- 发布期间保留旧 Bash 版本可回退入口；若新版本出现阻断故障，临时回切旧入口并发布修复版本。
